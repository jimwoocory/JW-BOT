# Lossless Store Sidecar Plan

## Goal

Define a Phase 1B storage contract for the AstrBot-native `lossless-claw-enhanced` integration in `JW-Bot`.

This phase should:

- preserve AstrBot's current `data_v4.db` conversation behavior
- add an inspectable sidecar store keyed by AstrBot `conversation_id`
- capture raw message lineage before any future summary compaction
- avoid invasive schema changes to AstrBot's primary database layer

## Why A Sidecar Store

AstrBot's current conversation model in [`po.py`](/Users/dianchi/JW-Bot/astrbot/core/db/po.py) stores the active chat state as a flat `content` list on `ConversationV2`.

That works well for normal operation, but it is too flat for:

- leaf-vs-summary lineage
- reversible compaction
- DAG summary tiers
- recovery and inspection tools
- compaction job bookkeeping

Phase 1B should therefore introduce a sidecar store instead of altering the main `ConversationV2` contract.

## Storage Shape

Recommended first implementation:

- a dedicated SQLite file under `data/`
- for example: `data/lossless_context.db`

Reasons:

1. clear ownership and easier rollback
2. no migration risk to AstrBot WebUI features that already assume `data_v4.db`
3. easier to inspect and back up during experimentation
4. easier to drop and rebuild if the schema changes in Phase 1C or 1D

## Keying Strategy

Every record should be keyed by AstrBot `conversation_id`.

That means:

- `ConversationV2.conversation_id` remains the session anchor
- the sidecar store does not invent a parallel conversation namespace
- future recall tools can always map back to AstrBot conversations directly

Useful secondary keys:

- monotonic sequence number per conversation
- item id for raw messages and summary nodes
- parent summary id for future DAG condensation

## Phase 1B Tables

### 1. `lossless_items`

Stores raw message items and later summary nodes in a shared item namespace.

Suggested columns:

- `item_id` text primary key
- `conversation_id` text not null
- `seq` integer not null
- `item_type` text not null
  - `raw_message`
  - `summary_leaf`
  - `summary_condensed`
- `role` text null
- `content_json` text not null
- `token_estimate` integer not null default `0`
- `source_start_seq` integer null
- `source_end_seq` integer null
- `created_at` text not null
- `status` text not null default `active`

Phase 1B usage:

- only `raw_message` rows are required initially
- summary rows are reserved for Phase 1C and 1D

### 2. `lossless_links`

Stores lineage between summary nodes and the raw or summary items they cover.

Suggested columns:

- `parent_item_id` text not null
- `child_item_id` text not null
- `link_type` text not null
  - `covers`
  - `condenses`
- `created_at` text not null

Phase 1B usage:

- table may be created now even if it stays empty until compaction work lands

### 3. `lossless_heads`

Stores per-conversation assembly checkpoints and fresh-tail boundaries.

Suggested columns:

- `conversation_id` text primary key
- `last_ingested_seq` integer not null default `0`
- `fresh_tail_start_seq` integer not null default `1`
- `last_compacted_seq` integer not null default `0`
- `active_summary_root_id` text null
- `updated_at` text not null

Purpose:

- lets the future assembler rebuild model-facing context deterministically
- keeps fresh-tail protection explicit instead of implicit

### 4. `lossless_jobs`

Tracks ingest and compaction operations for observability and crash recovery.

Suggested columns:

- `job_id` text primary key
- `conversation_id` text not null
- `job_type` text not null
  - `ingest`
  - `compact_leaf`
  - `condense`
- `status` text not null
  - `running`
  - `completed`
  - `failed`
- `started_at` text not null
- `finished_at` text null
- `error` text null
- `payload_json` text null

## Ingest Lifecycle

Phase 1B should only establish reliable ingest.

Recommended flow:

1. Read AstrBot conversation content from `ConversationV2.content`.
2. Compare with `lossless_heads.last_ingested_seq`.
3. Append any unseen messages into `lossless_items` as `raw_message`.
4. Update `lossless_heads.last_ingested_seq`.
5. Never mutate AstrBot's `ConversationV2.content` in this phase.

This makes Phase 1B observational first, not behavior-changing.

## Serialization Model

Compaction work later must not race with message ingest.

Recommended rule:

- maintain one async lock per `conversation_id`

Phase 1B only needs:

- `ingest(conversation_id, messages)`
- `get_state(conversation_id)`

Phase 1C and later can reuse the same per-conversation lock for compaction.

## Failure Model

If sidecar ingest fails:

- AstrBot should continue using the normal conversation path
- the sidecar error should be logged
- no user-facing failure should be introduced

This keeps the sidecar non-blocking while it matures.

## Initial Python Surface

Recommended first module:

- [`lossless_store.py`](/Users/dianchi/JW-Bot/astrbot/core/agent/context/lossless_store.py)

Recommended first class:

- `LosslessContextStore`

Recommended initial methods:

- `initialize()`
- `ingest_messages(conversation_id, messages)`
- `get_head(conversation_id)`
- `list_items(conversation_id, limit=...)`
- `record_job(...)`

The first implementation can use `aiosqlite` directly to keep this store isolated from AstrBot's larger SQLModel database abstraction.

## Integration Points

Phase 1B should be called from a narrow wrapper near the existing conversation/context boundary, not from many plugin entrypoints.

Best initial hook points:

- after conversation history is loaded for context assembly
- before future compression decisions run

Avoid in Phase 1B:

- WebUI integration
- plugin-level direct writes
- summary generation
- user-facing recall commands

## Phase 1B Exit Criteria

This phase is complete when:

1. a sidecar DB file is created automatically
2. each active AstrBot conversation can be mirrored into sidecar raw-message rows
3. repeated ingest is idempotent
4. a per-conversation head record is maintained
5. failures do not break normal AstrBot replies

## Follow-On Work

Once Phase 1B is stable:

1. Phase 1C can add summary leaf rows and fresh-tail compaction
2. Phase 1D can add DAG condensation and recall tooling

That keeps the migration incremental and keeps AstrBot as the only runtime host.
