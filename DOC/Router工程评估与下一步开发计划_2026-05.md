## 二、2026 年路由 / 意图分类文献综述

### 2.1 演化阶梯

业界 2026 年共识把 router/intent classifier 划成四层（[Label Your Data](https://labelyourdata.com/articles/machine-learning/intent-classification)、[Patronus AI Routing Best Practices](https://www.patronus.ai/ai-agent-development/ai-agent-routing)）：

| 层级 | 实现 | 代表延迟 | 适用 |
|------|------|---------|------|
| 1. Rule-based | 关键词 / 正则 / 命令前缀 | μs 级 | 高频确定性触发 |
| 2. Classical ML | TF-IDF + SVM / Random Forest | 毫秒级 | 有标签数据集 |
| 3. Fine-tuned Transformer | BERT / SetFit / MPNet | 数十毫秒 | 中等规模、稳定标签 |
| 4. LLM Prompt | GPT / Claude few-shot | 100ms – 数秒 | 标签开放、零样本 |

**生产实践共识**：单层不够。混合管线胜出——快速规则前置兜底 + 中速分类器主路 + LLM fallback 仅在不确定时调用。

> *"Hybrid approaches run a fast embedding match first and fall back to the LLM only when confidence is below a threshold, typically 0.85."* — [Patronus AI](https://www.patronus.ai/ai-agent-development/ai-agent-routing)

### 2.2 信心三段制（Three-Zone Routing）

来自 [Patronus AI](https://www.patronus.ai/ai-agent-development/ai-agent-routing)、[Prospeo](https://prospeo.io/s/intent-prediction)：

```
confidence ≥ T_high (0.85–0.95)  → 自动执行（可加二次确认）
T_low ≤ confidence < T_high      → 澄清提问（clarify）
confidence < T_low (≈0.50)       → 兜底菜单 / 人工 / safe default
```

每个意图的阈值**应单独调**，调整依据是：转移成本、处理时长、用户挫败感。

### 2.3 不确定性路由（Uncertainty-Based Routing）

学术里最具实操性的一篇是 [*Intent Detection in the Age of LLMs* (arXiv:2410.01627)](https://arxiv.org/html/2410.01627v1)：

- **结论**：用 SetFit + Monte Carlo Dropout 当主路，**只有不确定的样本走 LLM**——可以做到"在 LLM 原生准确率 2% 误差内、延迟降一半"。
- **OOS（out-of-scope）检测**：作者提出基于 LLM 隐层表示的两步法，OOS recall +5%，几乎零额外延迟。
- **关键洞察**：标签设计比模型选择更重要——细粒度、专属意图比粗大类更可靠。

更轻量的路径：[*Fast Intent Classification for LLM Routing via Statistical Analysis of Representations*](https://openreview.net/forum?id=UMuVvvIEvA)（2026 ICLR submission）展示**训练-free** 的统计分类器，利用首次 forward pass 的隐层特征即可分类、对 OOD 输入更鲁棒。

### 2.4 评测基准

| 名称 | 规模 | 维度 | 链接 |
|------|------|------|------|
| **RouterBench** | 405k 推理结果、64 任务、11 模型 | cost-quality 凸包、AIQ | [arXiv:2403.12031](https://arxiv.org/abs/2403.12031) |
| **RouteLLM** | 4 路由器、2 模型对（GPT-4 / Mixtral） | 成本节省 vs 质量 | [GitHub: lm-sys/RouteLLM](https://github.com/lm-sys/RouteLLM) |
| **RouterEval** | model-level scaling 视角 | 路由策略系统化对比 | [GitHub: RouterEval](https://github.com/MilkThink-Lab/RouterEval) |

> 重要区分：上述 benchmark 评的是**多模型路由（哪个 LLM 来答）**；DC-Agent 的 Router 评的是**意图路由（task / skill / conversation）**。两者度量框架可借鉴，但需要自建意图侧 ground truth。

### 2.5 主流失败模式与缓解（[Patronus](https://www.patronus.ai/ai-agent-development/ai-agent-routing)）

| 失败模式 | 表现 | 缓解 |
|---------|------|------|
| **模糊多意图** | "买信用卡 + 登录问题" | 多 label 输出 + 角色明确化 |
| **概念漂移** | 新业务/新词进来 | 持续监测准确率 + 周期 retrain |
| **上下文丢失** | 多轮对话中意图错判 | 把会话历史喂给分类器 |
| **级联失败** | 错误意图触发错误 agent → 错误工具 | 关键节点 LLM-judge 校验 |
| **越权调用** | 不该调用的 skill 被触发 | API permission 强约束 |

### 2.6 一句话提炼

> **2026 年的 router 主线：从"规则 / 单 LLM 二选一"走向"金字塔 + 不确定性 + 自带评测"——快速规则兜底，分类器担纲，LLM 只接不确定样本，整体闭环要可度量。**

## 三、DC-Agent Router 现状盘点

### 3.1 架构

源文件：[`astrbot/core/router.py`](../astrbot/core/router.py)、[`astrbot/core/pipeline/process_stage/router_stage.py`](../astrbot/core/pipeline/process_stage/router_stage.py)、[`astrbot/core/router_config.yaml`](../astrbot/core/router_config.yaml)。

```
incoming AstrMessageEvent
  ↓
RouterStage._check_hermes_escalation
  ↓ (SatisfactionDetector — 基于规则 + 信心 0.65/0.88 二级阈值)
  ├── 命中 → _handle_hermes_escalation → Hermes webhook 派发
  └── 未命中 ↓
RouterStage.route → IntentRouter.classify
  ↓
  ├── 1) RouterRule 匹配（pattern + keywords）
  │      最佳规则 confidence ≥ 0.75 → 直接返回
  ├── 2) LLM fallback（_classify_with_llm 调当前 provider.text_chat）
  │      返回 JSON → 若 confidence ≥ rule，胜出
  └── 3) 都不行 → default_intent("conversation", 0.4)
  ↓
分发：
  - category=task & workflow_kind → 创建 Harness 任务（不拦截事件，让 LLM 先答）
  - category=skill → 通过合成 message 激活对应 Star handler
  - 其他 → 落到普通 LLM 处理
```

### 3.2 配置内容（YAML 实际状态）

[`router_config.yaml`](../astrbot/core/router_config.yaml) 224 行：

- `fallback_threshold: 0.75`
- **task_intents**：5 条（`/task new`、Hermes 关键词、4 大 workflow_kind 的弱关键词）。注意 4 个 workflow_kind 关键词置信度都设成 **0.70**，**有意低于阈值**，目的是强制走 LLM 二次确认。
- **skill_intents**：~25 条，覆盖 dreamina（精确 + 模糊关键词两套）、github、google-workspace、notion、find-nearby、arxiv、powerpoint、research-paper-writing、linear、obsidian、hermes-agent。
- **llm_fallback.system_prompt**（行 184-224）：完整定义可用 workflow_kind / skill_name 白名单 + 输出 JSON schema + "除非非常确定，否则 confidence 不要 > 0.75" 自我克制。

### 3.3 LLM fallback 通路活性证据

- [`router_stage.py:35`](../astrbot/core/pipeline/process_stage/router_stage.py:35) `IntentRouter.from_yaml(..., llm_provider=self._classify_with_llm)`
- [`router_stage.py:449-481`](../astrbot/core/pipeline/process_stage/router_stage.py:449) `_classify_with_llm` 实际调 `plugin_context.get_using_provider(session_id).text_chat`
- [`router.py:178-179`](../astrbot/core/router.py:178) 双重短路保护（provider 为 None 或 prompt 为空时跳过）
- 单测覆盖：[`test_router_intents.py:786-980`](../tests/unit/test_router_intents.py:786) 覆盖 LLM fallback 命中、code-block JSON、阈值边界、provider 抛错容错

**Pytest**：`test_router_intents.py` + `test_router_harness_integration.py` = **107 passed** ✅。

**结论**：原 Harness 路线图所列的 Phase 0.4 任务（"Router LLM fallback 接通 provider"）实际已交付，**无需开新工**。

### 3.4 与 Harness 的协同

- Router 命中 task 类意图时**不拦截事件**（[router_stage.py:133](../astrbot/core/pipeline/process_stage/router_stage.py:133)）——任务创建但 LLM 仍正常作答；Harness 在后端跟。
- 若用户对回答不满意（[satisfaction.py SatisfactionDetector](../astrbot/core/harness/satisfaction.py)），Router 二次进来时 `_check_hermes_escalation` 命中，把已有任务派发给 Hermes 深度处理。
- 这是一个**渐进式的 escalation 模型**——LLM 先尝试，不行再升级，符合 Anthropic"轻 agent + 重 fallback"原则。

## 四、差距矩阵

| 维度 | DC-Agent | 2026 SOTA | 评级 | 关键差距 |
|------|----------|-----------|------|---------|
| 规则层 | YAML pattern/keywords | 同 | ✅ | 已具备 |
| 分类器层（中间） | **无** | SetFit / BERT / 嵌入 | 🔴 | 没有 ML 分类器，规则没命中就直接给 LLM 增加 100ms+ 延迟 |
| LLM fallback | 已接通，prompt 完整 | LLM zero-shot + ICL | ✅ | 可加 few-shot 示例提升 |
| 信心三段制 | 单阈值 0.75 | T_low / T_high 两段 | 🟡 | 没有 clarify 区——介于 0.5-0.75 的样本直接落 conversation |
| 不确定性 | 无 | MC Dropout / hidden-state stats | 🔴 | 完全没有 |
| 多意图（multi-label） | 单分类 | 多 agent 同时分发 | 🔴 | "营销 + 配图"只能选一 |
| 上下文 | 仅传 platform/session 元数据 | 历史会话喂入 | 🟡 | 上下文未注入分类提示 |
| Cost-aware 路由 | 单 provider | RouteLLM / RouterBench | 🔴 | 同一个 provider 答所有问题 |
| OOS 检测 | 默认 conversation 0.4 | LLM hidden-state 两步法 | 🟡 | 兜底简单但易把"未识别 task"误判为闲聊 |
| 失败模式遥测 | logger.info | route accuracy / clarify rate / drift | 🔴 | 没有 dashboard、没有滚动准确率 |
| 评测 | 107 单测 + 集成 | RouterBench / 自建意图基准 | 🟡 | 单测保正确性，缺 ground-truth 漂移监控 |
| 多轮一致性 | 每轮独立分类 | 状态机 + handoff | 🟡 | 同会话内多次分类无约束（可能反复跳类） |
| Skill 路由（多 skill） | 一对一 keyword | 嵌入相似度 / Top-K | 🟡 | 25 条规则手工维护，新增 skill 摩擦大 |
| LLM-judge 评审 | 无 | 关键意图二次校验 | 🔴 | 错路由级联问题无人捕捉 |

**结论**：DC-Agent Router 已经稳稳处于"Stage 1 + Stage 4 混合"，跳过了中间的 ML 分类器层；这正是 2026 年文献明确指出"应该补回来"的中段。

## 五、下一步开发方向

### 5.1 北极星

> **让 Router 成为"金字塔分层 + 信心三段制 + 自带评测"的可观测意图门户**——在不破坏现有规则/LLM 双层的前提下，补齐中段嵌入分类器、把单阈值升级为三段制、把所有路由决策落到 OTEL trace + 滚动准确率 dashboard。

### 5.2 设计原则

1. **不替换、只插层**：嵌入分类器作为 rule 与 LLM fallback 之间的中段，规则命中仍最快。
2. **信心三段化**：每个意图独立配置 T_low/T_high；中间区强制 clarify。
3. **决策可追溯**：每次 classify 必落 trace（matched_by、confidence、latency、provider tokens）。
4. **Eval 优先于 Feature**：先有 router benchmark v0，再上新 skill。
5. **意图标签治理**：新增 skill 必带 ≥ 5 条 ground-truth 样本入 benchmark。

### 5.3 全栈分层与五条主线

> Router 不是单文件后端任务——它需要**后端 + ML/嵌入 + 聊天端 UI + 管理面板 + DevOps + Skill 自服务**六层全栈协同。

| 层 | 触达点 | 关键产物 |
|---|--------|---------|
| **后端 (Backend)** | `astrbot/core/router*.py`、`router_stage.py` | classify span、三段制、嵌入中段、multi-label |
| **ML / Embedding** | 新增 `astrbot/core/router/embedding/` | 中心向量库 + cosine 检索 + 离线 retrain |
| **聊天端 UI (QQ/Feishu/WebChat)** | `MessageEventResult` 扩展、t2i 卡片 | clarify 菜单卡、skill quick-pick chips、"路由理由"折叠提示 |

| **DevOps / Infra** | OTEL collector、jsonl rotation、A/B canary | trace 上报、5% 灰度路由、配置热加载（zero-downtime reload） |

## 七、与 Harness 路线图的衔接

| 同步点 | 说明 |
|-------|------|
| **OTEL trace 共享** | Router R0.1 与 Harness Phase 3.1 共用同一 trace_id，端到端可串联 |

| **LLM-judge 复用** | R3.4 与 Harness Phase 3.4 共用同一 judge prompt/provider，节省调度成本 |

**建议总顺序**（合并两份路线图，按数据流方向 Router → Harness）：

| 周 | Router | Harness | 备注 |
|---|--------|---------|------|
| W1-1.5 | R0（可观测 + bench + 实时流） | Phase 0.3（仅剩的 Hermes Bridge） | Router 先铺 trace，Harness 后端复用 |
| W1.5-3 | R1（三段制 + clarify 三端） | Phase 1.1-1.2（多轮 loop + Planner） | clarify 卡片可作为 Planner 的 user touch-point |
| W3-5 | R2（嵌入中段 + 聚类视图） ‖ Harness Phase 2（上下文记忆） | 并行 | Router 嵌入与 Harness mem0 复用同一 embedding 服务 |
| W5-7 | R3（多意图 + judge） ‖ Harness Phase 3（评测可观测） | 并行 | judge 与 OctoBench-DC 共享 prompt + provider |
| W7-8 | R4（漂移 + 标注台） ‖ Harness Phase 4（前端 + 安全） | 并行 | 漂移热力 + Harness 时间线整合到一个 dashboard |
| W8-9.5 | R5（Rule Editor + Skill Marketplace + 灰度） | （收尾） | 把整套路由治理落到管理面板 |

**总共 9.5 周端到端**完成上下游全栈升级（235h Router + 265h Harness ≈ 500h，可由 1 后端 + 1 前端 + 0.5 ML + Claude Code 协助分担）。
