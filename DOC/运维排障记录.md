## 2026-04-23 | QQ Bot 回复"OpenClaw 服务未运行"

### 现象
员工向 QQ Bot 发消息，Bot 回复：
> ⚠️ [系统提示] 无法完成指令。状态：请检查本地 OpenClaw Core 服务是否正常运行（端口 18789）...

### 根本原因
两个独立问题叠加：

**问题 1：Persona 的 custom_error_message 残留旧内容**
`data/data_v4.db` 的 `personas` 表中，3 个 Persona 的 `custom_error_message` 字段存有 OpenClaw 相关错误提示。当 LLM pipeline 抛出任何异常时，AstrBot 会返回该字段内容而非通用错误。

修复：
```sql
UPDATE personas
SET custom_error_message = '抱歉，处理您的请求时遇到了问题，请稍后再试。'
WHERE id IN (1, 2, 3);
```

**问题 2：会话历史损坏（tool role 孤儿消息）**
LLM 请求时报 `400 BadRequestError`：
> messages with role 'tool' must be a response to a preceeding message with 'tool_calls'

会话历史（264 条消息，99202 tokens）中存在孤儿 tool 消息，lossless compressor 也因此无法压缩，形成死循环。

修复（清空损坏会话）：
```sql
-- data/data_v4.db
UPDATE conversations
SET content = '[]', token_usage = 0
WHERE inner_conversation_id = 7;  -- 对应 conversation_id: fed66aa6-...

-- data/lossless_context.db
DELETE FROM lossless_items WHERE conversation_id = 'fed66aa6-7e41-491f-bcc2-d6dd2e920a39';
DELETE FROM lossless_heads WHERE conversation_id = 'fed66aa6-7e41-491f-bcc2-d6dd2e920a39';
```

### 经验
- persona `custom_error_message` 会在 LLM 任何异常时返回，要保持通用措辞
- 会话历史损坏后，lossless 压缩和正常请求都会失败；清空 content 是最快修复手段
- 清空后需同步清除 `lossless_context.db` 中的对应记录

## 2026-04-23 | QQ Bot 回复成功但用户收不到消息（Connection reset）

### 现象
AstrBot 日志显示 LLM 生成了完整回复，但出现：
```
ClientConnectorError: Cannot connect to host api.sgroup.qq.com:443
  → Connection reset by peer
```

### 根本原因
路由器（192.168.8.1）上的 Surge TUN 将 `api.sgroup.qq.com` 和 `multimedia.nt.qq.com.cn` 的流量走了不稳定的代理节点，偶发连接 reset。

确认特征：DNS 返回 `198.18.x.x`（Surge fake-IP 段），说明流量被 TUN 拦截。

修复：在路由器 Surge 规则中为 QQ API 域名添加 DIRECT 规则：
```
DOMAIN-SUFFIX,sgroup.qq.com,DIRECT
DOMAIN-SUFFIX,nt.qq.com.cn,DIRECT
```

### 经验
- `198.18.x.x` IP 段是 Surge fake-IP 的标志，出现即说明流量经过路由器 TUN
- 腾讯/QQ 域名直连更稳定，不需要走代理
- 飞书域名同理（open.feishu.cn），如出现 DNS 失败也考虑加 DIRECT

## 2026-04-23 | Dashboard Bot 卡片显示"1个错误"红色徽标

### 现象
AstrBot Dashboard 的 Bot 管理页面，4 个 Bot 卡片各显示红色"1个错误"。

### 根本原因
AstrBot 启动后立即尝试连接各平台 Websocket，此时 Surge TUN 尚未完全初始化，DNS 解析失败，连接报错。重试后成功，但错误计数已记录。

修复：在 `start-astrbot.sh` 启动脚本中加入 30 秒等待：
```bash
echo "[start-astrbot] Waiting 30s for Surge TUN to initialize..."
sleep 30
```

### 经验
- 错误徽标是累计计数，不是实时状态；连接成功后徽标不会自动清零
- 根本解法是给 TUN 足够的初始化时间，而非修改 AstrBot 代码

## 通用排障命令

### 查看实时日志
```bash
tail -f /Users/dianchi/JW-Bot/logs/launchd.out.log | grep -v "^\s*$"
```

### 查找错误
```bash
grep -E "ERRO|Exception|BadRequest|Connection reset" /Users/dianchi/JW-Bot/logs/launchd.out.log | tail -30
```

### 查看 Harness 任务状态
```bash
sqlite3 data/harness.db "SELECT task_id, status, title, updated_at FROM harness_tasks ORDER BY updated_at DESC LIMIT 10;"
```

### 查看最新记忆记录
```bash
sqlite3 data/harness_memory.db "SELECT session_id, substr(summary,1,80), created_at FROM harness_memories ORDER BY created_at DESC LIMIT 5;"
```

### 查看会话 token 用量
```bash
sqlite3 data/data_v4.db "SELECT inner_conversation_id, token_usage, user_id FROM conversations ORDER BY token_usage DESC LIMIT 5;"
```

### 检查 QQ API 连通性
```bash
curl -v --max-time 10 https://api.sgroup.qq.com 2>&1 | grep -E "Connected|reset|SSL|error"
# 198.18.x.x = Surge fake-IP（流量经过 TUN）
# 真实 IP = 直连
```
