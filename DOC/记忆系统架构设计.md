# JW-Bot 记忆系统架构设计

> 参考：lossless-claw-enhanced（短期压缩）、memory-lancedb-pro（长期语义召回）

---

## 一、两层记忆架构

```
短期记忆（Short-term）              长期记忆（Long-term）
━━━━━━━━━━━━━━━━━━━━━━              ━━━━━━━━━━━━━━━━━━━━━━
AstrBot 对话窗口                    Harness Task Memory
  └─ 最近 N 轮对话                    └─ 已完成任务摘要
  └─ AstrBot 原生管理                 └─ SQLite: harness_memory.db
  └─ 无需额外开发 ✅                  └─ 写入 + 注入已实现 ✅
```

---

## 二、短期记忆（已有，无需开发）

AstrBot 原生维护对话上下文（ConversationContext），自动保存最近 N 轮对话到数据库，
每次 LLM 调用时随 `req.contexts` 传入。

设计参考：lossless-claw-enhanced 的 **protect-last-N** 理念 — 最近轮次全量保护，
更早的轮次按 token 预算决策是否压缩。

## 三、长期记忆（Harness Task Memory）

### 3.1 写入路径

```
Hermes 执行完毕
  → POST http://localhost:8645/task_result
  → hermes_bridge._handle_hermes_response()
  → hermes_bridge._complete_harness_task()
  → engine.complete_task(result={"summary": ..., "source": "hermes"})
  → HarnessEngine._maybe_promote_memory()
  → HarnessMemoryPromoter.promote_from_task()
  → HarnessMemoryStore.create_memory()          ← 写入 harness_memory.db
```

每条记忆字段：

| 字段 | 内容 |
|------|------|
| memory_id | UUID |
| session_id | 员工的 unified_msg_origin（会话隔离） |
| task_id | 关联的 Harness 任务 ID |
| domain | marketing / project / delivery / approval |
| title | 任务标题（含 workflow_kind 前缀） |
| summary | 任务结果摘要（前 200 字） |
| created_at | UTC 时间戳 |

### 3.2 读取路径（注入 LLM）

```
员工发消息 → AstrBot LLM 即将调用
  → @filter.on_llm_request() 钩子触发
  → hermes_bridge.inject_harness_memory()
  → HarnessMemoryStore.list_for_session()     ← 取候选（最多 limit×3 条）
  → _score_memories(message, candidates)      ← 混合评分排序
  → 取 top-N 注入 req.system_prompt           ← LLM 看到历史记忆
```

### 3.3 记忆评分算法

参考 memory-lancedb-pro 的 **hybrid recall** 思路（简化版，无向量 DB）：

```
score = 0.5 × recency + 0.5 × domain_relevance

recency         指数衰减，半衰期 7 天
                recency = exp(-ln2 × days_old / 7)

domain_relevance CJK n-gram 子串匹配（2~4 字 n-gram）
                 提取消息所有 2~4 字子串 → 检查命中记忆文本的比例
                 domain_score = min(1.0, hits / len(ngrams) × 5)
```

**评分示例（验证通过）：**

| 消息 | 记忆 1（1天前） | 记忆 2（14天前） | 记忆 3（30天前） |
|------|----------------|-----------------|-----------------|
| 营销策划/抖音 | **0.796**（命中） | 0.372 | 0.026 |
| 供应链情况 | 0.453 | **0.438**（命中） | 0.026 |
| 预算审批通过 | 0.453 | 0.372 | **0.526**（命中，domain 压制时间） |

### 3.4 注入格式

```
# 工作记忆（历史任务摘要，供回答参考，无需向用户重复）
- [2026-04-25][marketing] 营销策划 | 抖音推广方案：聚焦抖音+私域，KPI=月GMV翻倍
- [2026-04-26][project] 项目跟进 | 供应链风险：供应链延期，建议提前备货30天
```

## 四、配置项（hermes_bridge config section）

```yaml
hermes_bridge:
  memory_inject_enabled: true   # 是否启用长期记忆注入
  memory_inject_limit: 5        # 每次最多注入几条记忆
```

## 五、未来扩展方向

| 阶段 | 扩展 | 说明 |
|------|------|------|
| Phase 2 | 向量嵌入 | 换用 LanceDB 做语义搜索（需安装 lancedb） |
| Phase 2 | 跨 session 记忆 | 同一员工在不同 QQ 群的记忆合并 |
| Phase 3 | 记忆压缩 | 对话窗口满时 LLM 分层压缩（lossless-claw 方案） |
| Phase 3 | 记忆衰减管理 | Weibull 衰减 + 定期清理过期低分记忆 |
| Phase 3 | 主动记忆提醒 | 检测到相关任务时主动告知员工上次结论 |
