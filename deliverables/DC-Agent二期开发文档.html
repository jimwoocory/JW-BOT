二期定位：把一期飞书机器人从“能聊、能生成”升级为“能跟项目、能找资料、能沉淀结果”的项目执行助手。第一阶段不做泛化大而全能力，集中解决项目群信息分散、任务容易遗漏、客户资料难查、方案初稿耗时四类问题。
数据依据：2026-05-06 飞书问卷 11 份有效答复；主要覆盖品宣部、中台运营、执行运营、财务部及其他岗位。

一、问卷结论与开发定向

91%选择任务提取和提醒
82%选择项目群聊总结
73%希望私聊查询或生成
82%希望读取飞书文档
定版判断：员工最急需的不是群里自动抢答，而是可靠的项目助理能力。二期应优先做“项目群聊总结 + 任务提取提醒 + 资料查询 + 方案生成”四件事，默认通过私聊、按钮、固定指令或群内 @ 触发，避免机器人主动干扰群聊。
问卷信号 占比 开发含义
任务提取和提醒 10/11 必须作为 P0，围绕负责人、事项、截止时间、状态提醒形成闭环。
项目群聊总结 9/11 必须作为 P0，用于沉淀客户反馈、老板要求、执行问题和阶段结论。
公司/客户/项目资料查询 5/11 作为 P0/P1 衔接能力，优先接飞书文档、飞书表格、客户资料和项目资料。
飞书群 @ 问答 5/11 保留为触发入口，但不做默认自动回应。

二、二期开发目标

业务目标

  • 减少项目群信息遗漏，形成可复盘的项目记录。
  • 把任务、负责人、截止时间从聊天内容中自动提取出来。
  • 让员工能快速查到客户、项目、历史方案和执行资料。
  • 提升方案、文案、脚本和提案大纲初稿产出效率。

产品原则

  • 默认低打扰：私聊、按钮、固定指令优先。
  • 群内可控：需要 @ 或明确指令才回应。
  • 准确优先：输出应尽量说明来源或依据。
  • 先垂直再扩展：优先服务汽车客户项目。

三、二期功能范围

模块 优先级 功能说明 验收标准
项目群聊总结 P0 读取指定项目群近期讨论，整理客户反馈、老板要求、执行问题、阶段结论。 群内 @DC-Agent 后，可生成结构化总结；总结包含重点、问题、结论、待办。
任务提取和提醒 P0 从群聊或私聊中提取任务、负责人、截止时间和状态，生成任务台账。 至少能识别事项、负责人、截止时间；支持按负责人查看待办并定时提醒。
资料查询入口 P0 优先接入飞书文档、飞书表格、客户资料、项目资料，支持历史方案查询。 能按客户名、项目名、关键词检索资料；回答中标注来源或资料链接。
内容与方案生成 P1 生成客户提案大纲、短视频脚本、小红书文案、口播文案和活动方案。 员工输入客户需求后，能生成可编辑初稿，并支持继续修改。
项目进度看板 P1 基于任务台账汇总项目状态、负责人进展、逾期风险和待确认事项。 支持查看“今天要做什么”“项目卡在哪里”“谁还有待办”。
热点与竞品监测 P2 面向品宣部，整理平台热点、竞品动态、私域负面信息和选题建议。 作为后置专项，不阻塞二期主线上线。

四、技术实现拆分

系统层 建设内容 说明
AstrBot 前台 飞书群/私聊入口、@ 触发、固定指令、按钮菜单 继续作为员工交互入口，避免让 Hermes 直接面对员工。
Router 意图识别：总结、查资料、写方案、提任务、看进度 把员工自然语言分发到对应模块，减少误触发。
Harness 任务生命周期、状态记录、待办台账、记忆提升 用于沉淀任务过程，支撑后续复盘和项目看板。
资料索引 飞书文档、飞书表格、客户资料、项目资料 先做小范围白名单资料源，确保答案可追溯。
LLM 生成 总结、提案、脚本、文案、方案框架 所有生成类结果都要支持继续修改和补充约束。
Hermes 后台 复杂任务深度处理预留 二期不强制做自动升级闭环；保留人工或指令触发深挖能力。

五、开发时间轴

2026-05-07 至 2026-05-10

确认二期范围、P0/P1/P2 边界、飞书机器人使用规则、首批项目群和首批客户资料源。输出详细字段表、指令表、权限清单和验收样例。

2026-05-11 至 2026-05-17
阶段 1：项目群聊总结 MVP

完成群聊摘要模板、客户反馈提取、老板要求提取、阶段结论输出。支持群内 @DC-Agent 总结近期讨论。

2026-05-18 至 2026-05-24
阶段 2：任务提取与提醒台账

完成任务识别、负责人识别、截止时间识别、任务状态记录、按负责人查看待办、每日提醒规则。优先落地飞书表格或多维表格台账。

2026-05-25 至 2026-05-31
阶段 3：资料查询与来源标注

接入首批飞书文档、飞书表格、客户项目资料；支持按客户、项目、关键词查询；回答中展示资料来源、链接或引用依据。

2026-06-01 至 2026-06-07
阶段 4：内容与方案生成增强

围绕品宣和中台运营，完成客户提案大纲、短视频脚本、口播文案、小红书/朋友圈文案、活动方案框架生成与二次修改。

2026-06-08 至 2026-06-14
阶段 5：联调、灰度与验收

选择 2 至 3 个项目群灰度测试，验证总结准确率、任务提取准确率、提醒可用性、资料查询命中率和员工满意度。修复高频问题。

2026-06-15
二期正式上线建议日

发布给品宣部、中台运营、执行运营试运行；财务对账、私域负面监控、自动深挖作为后续专项立项。

六、验收标准

验收项 目标 通过标准
群聊总结可用性 P0 指定项目群内 @DC-Agent 后，可输出结构化总结，人工评估可用率达到 80% 以上。
任务提取准确性 P0 从测试群聊中提取任务、负责人、时间，核心字段准确率达到 75% 以上。
提醒闭环 P0 可按负责人生成待办，并在设定时间发送提醒。
资料查询命中 P0 对首批客户/项目资料，能返回来源明确的答案或链接。
生成内容可编辑 P1 提案大纲、脚本、文案可按员工补充要求继续修改。
低打扰策略 P0 群聊中默认不主动抢答，仅在 @、指令或已配置流程中回应。

七、暂不纳入二期主线

后置专项

  • 财务流水、发票、报销单自动匹配对账。
  • 私域社群本竞品负面信息监控与话术生成。
  • 全网热点/竞品自动采集的长期监测系统。
  • Hermes 满意度检测后自动升级深挖完整闭环。

不建议做法

  • 不做群里自动抢答，避免干扰项目沟通。
  • 不一次性接入所有文件和知识库，先做白名单资料源。
  • 不把图片/视频生成作为二期主线，只保留现有能力。
  • 不先做大而全的 AI 助手，先做可验收的项目执行助手。

角色 职责

九、最终定版

DC-Agent 二期开发方向正式定为：项目执行助手。
交付主线为：项目群聊总结、任务提取提醒、资料查询、方案生成。上线节奏按 6 周推进，2026-06-15 建议正式进入部门试运行。
